# CUA-Lark：基于多模态大模型的飞书桌面端智能操作代理

---

## 一、技术方案

### 1.1 总体架构

系统采用 **感知-推理-执行-验证** 四节点循环架构：

```
用户输入自然语言指令
         │
         ▼
  ┌──────────────────────────────────────────────────┐
  │              Agent 主循环 (Loop)                  │
  │                                                  │
  │   ┌────────────┐                                 │
  │   │  ① 感知层   │  截取飞书窗口 + 叠加网格编号     │
  │   │ Perception  │  + 界面状态识别                 │
  │   └─────┬──────┘                                 │
  │         ▼                                        │
  │   ┌────────────┐                                 │
  │   │  ② 推理层   │  截图 + 任务目标 + 操作历史       │
  │   │ Reasoning   │  → GPT-4o → 结构化动作指令       │
  │   └─────┬──────┘                                 │
  │         ▼                                        │
  │   ┌────────────┐                                 │
  │   │  ③ 执行层   │  解析动作 JSON → PyAutoGUI       │
  │   │ Execution   │  模拟鼠标点击 / 键盘输入          │
  │   └─────┬──────┘                                 │
  │         ▼                                        │
  │   ┌────────────┐                                 │
  │   │  ④ 验证层   │  对比操作前后截图 → 判断是否成功   │
  │   │ Validation  │  失败则触发重试或纠错             │
  │   └─────┬──────┘                                 │
  │         ▼                                        │
  │   回到 ① 重新截取                                 │
  └──────────────────────────────────────────────────┘
         │
         ▼
    任务完成 / 超时退出
```

### 1.2 感知层（Perception）

感知层负责获取飞书桌面端的当前屏幕状态。

**核心职责：**

- 截取飞书窗口区域（而非全屏，减少干扰信息）
- 在截图上叠加 Set-of-Mark 网格编号
- 记录当前截图的分辨率和窗口位置信息
- 可选：通过 OCR 提取截图中的文字信息作为辅助

**Set-of-Mark 网格定位方法：**

直接让视觉模型输出像素坐标的误差极大（±50~100px），原因是多模态模型对像素级精度的感知能力有限。本课题采用 Set-of-Mark (SoM) 方法，在截图上叠加带编号的网格，将"精确坐标回归"降维为"离散分类选择"，大幅降低视觉模型的推理难度。

```
 操作前：让模型猜坐标              操作后：让模型选格子
┌──────────────┐             ┌───┬───┬───┬───┐
│              │             │ 1 │ 2 │ 3 │ 4 │
│   [搜索框]    │    ──→     ├───┼───┼───┼───┤
│              │             │ 5 │ 6 │ 7 │ 8 │
│  [联系人列表] │             ├───┼───┼───┼───┤
│              │             │ 9 │10 │11 │12 │
└──────────────┘             └───┴───┴───┴───┘

 模型输出：x=580, y=230       模型输出：grid=7
  （误差 ±80px）              （误差 ±5%，稳定）
```

### 1.3 推理层（Reasoning）

推理层是系统的"大脑"，负责理解用户意图并生成操作指令。

**核心职责：**

- 接收感知层的截图（含网格编号）和用户的自然语言指令
- 调用 GPT-4o Vision API 进行视觉理解和推理
- 输出结构化的 JSON 动作指令
- 支持任务规划：将复杂指令拆解为有序子步骤

**结构化动作输出：**

| 动作类型 | JSON 格式 | 用途 |
|----------|-----------|------|
| click | `{"action":"click","grid":7,"reason":"..."}` | 点击界面元素 |
| paste | `{"action":"paste","text":"中文","reason":"..."}` | 粘贴文本（含中文） |
| type | `{"action":"type","text":"english","reason":"..."}` | 逐字输入英文 |
| hotkey | `{"action":"hotkey","keys":["cmd","k"],"reason":"..."}` | 快捷键组合 |
| press | `{"action":"press","key":"enter","reason":"..."}` | 按下单个键 |
| wait | `{"action":"wait","seconds":2,"reason":"..."}` | 等待界面响应 |
| done | `{"action":"done","reason":"任务已完成"}` | 标记任务结束 |

**系统提示词设计：**

```
你是一个桌面操作智能体（Computer Use Agent）。
当前正在操控的是飞书桌面客户端。

你的任务是根据用户的自然语言指令，分析屏幕截图，逐步完成目标。

规则：
1. 截图上有红色编号网格，用 grid 编号定位元素
2. 每次只返回一个动作的 JSON，不要多余内容
3. 中文内容使用 paste 动作（通过剪贴板粘贴）
4. 优先使用快捷键 Cmd+K / Ctrl+K 打开搜索（比点击更可靠）
5. 操作之间要有耐心，界面需要时间响应
6. 不确定时选择 wait，不要乱点
```

### 1.4 执行层（Execution）

执行层负责将推理层输出的结构化动作指令转化为实际的鼠标键盘操作。

**核心职责：**

- 解析 JSON 动作指令
- 调用 PyAutoGUI 模拟鼠标点击、键盘输入
- 处理中文输入（通过剪贴板粘贴方案）
- 管理飞书专属快捷键映射

**中文输入处理：**

PyAutoGUI 的 `typewrite()` 函数不支持中文字符。采用剪贴板粘贴方案：

```
用户指令含中文
      ↓
pyperclip.copy(中文文本)
      ↓
pyautogui.hotkey('Cmd+V')  /  pyautogui.hotkey('Ctrl+V')
      ↓
飞书输入框中出现中文
```

### 1.5 验证层（Validation）

验证层负责判断每步操作是否成功执行，并在失败时触发纠错机制。

**核心职责：**

- 操作前后各截取一次屏幕
- 通过视觉模型对比前后截图，判断操作是否成功
- 失败时根据错误类型选择策略：重试、回退、换路径
- 支持最大重试次数和超时控制

**纠错策略：**

| 错误类型 | 策略 |
|----------|------|
| 点击位置偏移 | 重新截屏定位，更换目标格子 |
| 界面未响应 | 等待后重试 |
| 弹窗干扰 | 先关闭弹窗，再继续原任务 |
| 路径走不通 | 回退到上一步，尝试替代路径 |
| 超过最大重试 | 终止任务并报告失败原因 |

### 1.6 辅助模块

#### 记忆模块（Memory）

| 类型 | 说明 | 实现方式 |
|------|------|----------|
| 短期记忆 | 当前任务的操作历史，避免重复操作 | 内存列表 |
| 长期记忆 | 跨任务的经验积累，复用成功路径 | SQLite 持久化 |
| 上下文管理 | 将记忆转化为推理可用的文本上下文 | 文本拼接 |

#### 知识库（Knowledge）

| 内容 | 说明 |
|------|------|
| 飞书模块定义 | 消息、日历、文档、审批、多维表格等模块的功能描述 |
| 界面元素特征 | 飞书各模块中常见 UI 元素的视觉特征描述 |
| 操作路径库 | 常见任务的标准操作路径（如"发消息"的步骤序列） |

#### 安全模块（Safety）

| 能力 | 说明 |
|------|------|
| 危险操作检测 | 识别删除、发送敏感信息等高风险操作 |
| 用户确认机制 | 危险操作前暂停执行，请求用户确认 |
| 操作回滚 | 支持撤销最近一步操作 |

### 1.7 技术栈

| 层次 | 技术选型 | 选型理由 |
|------|----------|----------|----------|
| 视觉推理 | Qwen-VL-Max (阿里云百炼) | 多模态模型，支持图像输入和推理 |
| 屏幕截取 | mss | 高性能跨平台截屏库 |
| 图像处理 | Pillow | 叠加网格标记 |
| 键鼠控制 (Windows) | PyAutoGUI + pywin32 | 鼠标键盘模拟 + 窗口管理 |
| 键鼠控制 (macOS) | Quartz + pyobjc | 原生 macOS API |
| 中文输入 | pyperclip | 剪贴板操作，解决中文输入问题 |
| OCR 辅助 | EasyOCR / PaddleOCR | 可选的文字识别辅助定位 |
| 数据持久化 | SQLite | 长期记忆和知识库存储 |
| 终端美化 | colorama | 彩色日志输出 |
| 环境管理 | python-dotenv | API Key 安全管理 |
| 编程语言 | Python 3.10+ | 生态成熟，开发效率高 |

## 二、项目框架

```
CUA-Lark/
│
├── src/
│   ├── app/                     # 应用层：agent 主循环、CLI 入口
│   └── platforms/               # 平台适配层
│       ├── common/              # 跨平台共享逻辑：网格、截图叠加
│       ├── windows/             # Windows 专属实现：截屏、点击、输入、快捷键
│       └── macos/               # macOS 专属实现：截屏、点击、输入、快捷键
│
├── .env                         # 环境变量
├── requirements.txt
├── requirements-macos.txt
├── environment.yml
├── pyproject.toml
├── Makefile
├── captures/                    # 调试截图与运行结果
├── LICENSE
└── README.md
```

### 当前分层说明

- `app/` 只负责任务编排，不直接写平台判断逻辑。
- `platforms/common/` 放所有跨平台可复用的能力，避免 macOS 和 Windows 重复实现同一套网格换算与叠图逻辑。
- `platforms/windows/` 与 `platforms/macos/` 各自封装系统 API，后续如果某些能力继续分叉，目录会自然承接。

### 模块间依赖关系

```
python cua-lark.py / cua-lark CLI
  │
  ▼
app/agent.py
  │
  ├──► platforms/__init__.py        当前操作系统选择器
  │         │
  │         ├──► platforms/windows/ Windows 实现
  │         └──► platforms/macos/   macOS 实现
  │
  └──► platforms/common/            共享网格与截图辅助逻辑
```

## 三、安装与运行

### 3.1 Windows 系统

```bash
# 1. 创建 conda 环境
conda env create -f environment.yml
conda activate cua-lark

# 2. 配置 API Key（编辑 .env 文件）
DASHSCOPE_API_KEY=your_api_key_here

# 3. 运行
python cua-lark.py 帮我给游畅发送你好
```

### 3.2 macOS 系统

```bash
# 1. 创建 conda 环境
conda env create -f environment.yml
conda activate cua-lark

# 2. 配置 API Key（编辑 .env 文件）
DASHSCOPE_API_KEY=your_api_key_here

# 3. 运行（统一入口，自动识别 macOS）
python cua-lark.py 帮我给游畅发送你好
```

**macOS 权限说明：**
需要在 `系统设置 > 隐私与安全性` 中授予以下权限：
- **辅助功能**：控制键盘和鼠标
- **屏幕录制**：截取屏幕内容
- **自动化**：控制其他应用

### 3.3 依赖说明

| 系统 | 依赖文件 |
|------|----------|
| Windows | `requirements.txt` |
| macOS | `requirements-macos.txt` |
